# specmate 愿景与定位

> 内部文档，不在公开仓库中。等实现出来再说。

## 核心命题

AI 模型能力持续提升，纯"知识查询"层（查规范、查示例）最容易被淘汰——当模型本身就能理解 BSV 语法和错误码时，查参考就变成了冗余。

**不会被淘汰的是流程本身。** 模型再强，也需要编排层来管理：
- 什么时候做什么（开发阶段）
- 踩过的坑怎么沉淀（项目记忆）
- 项目特定的约束怎么传递

## 三层架构

| 层 | 角色 | 举例 | 抗淘汰性 |
|---|---|---|---|
| **流程编排** | 定义怎么开发 | 预检 → 编译 → 学习 → 迭代 | 模型越强越需要编排 |
| **项目记忆** | 记住踩过的坑 | capture/resolve 私有知识库 | 模型不知道你的项目历史 |
| **领域知识** | BSV/硬件规范 | 参考文档、错误解释、示例 | 模型能力提升后价值递减 |

**核心投入在前两层。第三层是基础，不是唯一卖点。**

## 闭环开发流程

```
Agent 写 BSV 代码
  → specmate 预检（高精度规则，不搞大面积正则）
  → bsc 编译（Agent 自己调用）
  → specmate 匹配知识库 → 解释错误 + 建议修复
  → Agent 修复
  → specmate 学习入库（capture/resolve）
  → 下次同类问题秒出方案
```

specmate 不做 EDA 工具，做的是 **"AI 使用 EDA 工具的编排层"**。

## 从 BSV 到硬件设计

BSV 是入口，但整个硬件开发流程都可以套这个模式：

- RTL 设计（当前）
- 验证（testbench 生成 + 覆盖率反馈）
- 综合（时序报告解读 + 优化建议）
- 跨时钟域（CDC 检查 + 约束生成）

每个领域都可以套同一套"预检 → 执行 → 学习 → 记忆"的闭环。不做工具本身，做工具的编排者和经验的沉淀者。

## 不做的事

- 不做神经网络/模型训练（不是 AI 公司）
- 不做 EDA 工具替代（Synopsys/Cadence 的活）
- 不做通用编程助手（不跟 Copilot/Cursor 卷）
- 不做一次性知识库（模型自己会记住的东西不投入）

## 一句话

**让用 specmate 的团队和不用的是两种效率——不是因为它知道更多 BSV 语法，而是因为它能让 AI 越写越准、项目越积累越聪明。**
