# G0005 — no_implicit_conditions / fire_when_enabled 断言失败

## 现象

```
Error: "Top.bsv", line 96, column 10: (G0005)
  The assertion `no_implicit_conditions' failed for rule `rl_start_transfer'
```

或

```
Error: "Top.bsv", line 96, column 10: (G0005)
  The assertion `fire_when_enabled' failed for rule `RL_rl_start_transfer'
  because it is blocked by rule start in the scheduler
```

## 原因

1. `no_implicit_conditions`：rule 的条件中缺少 body 里方法调用的隐式 guard。例如 `resp_fifo.enq()` 需要 `notFull`，但 rule 条件中没写。
2. `fire_when_enabled`：rule 的条件为真时，调度器无法让 rule 触发——因为 method 和 rule 共享同一 FIFO（如 method 做 enq、rule 做 deq）导致调度冲突。

## 解决方案

推荐方案：**去掉这些属性**。编译器的默认调度足以处理大多数场景。

```bsv
// 错误 ❌
(* fire_when_enabled, no_implicit_conditions *)
rule my_rule (st == 1);
    resp_fifo.enq(data);  // enq 的隐式 guard notFull 未在条件中
endrule

// 正确 ✅（最简单）
rule my_rule (st == 1 && resp_fifo.notFull);
    resp_fifo.enq(data);
endrule

// 或直接去掉属性，让编译器处理
rule my_rule (st == 1);
    resp_fifo.enq(data);
endrule
```

如果确实需要 method/rule 并行访问同一 FIFO：
```bsv
(* descending_urgency = "my_method, my_rule" *)
```

## 规则

> `no_implicit_conditions` 要求 rule 的 CAN_FIRE 完全等于显式条件——开发中很难满足。除非在严格验证调度，否则不建议用此属性。
