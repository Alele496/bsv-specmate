# G0004 — Rule 内并行写冲突 (×2)

## 子情况 1：同一寄存器重复写入

**bsc 输出**：
```
Error: "...", line 77, column 10: (G0004)
  Rule `RL_dac_rl_spi' uses methods that conflict in parallel:
    dac_busy_r.write(...)
  and
    dac_busy_r.write(...)
```

**原因**：同一 rule 内对同一寄存器编写了两次写入。case 的 default 分支即使不执行也会触发。

**解决**：
```bsv
// 错误
rule rl_spi;
    busy_r <= True;
    case (cmd_type)
        WRITE: begin busy_r <= False; end
        default: begin busy_r <= False; end
    endcase
endrule

// 正确：删除 default 分支
rule rl_spi;
    case (cmd_type)
        WRITE: begin busy_r <= False; end
    endcase
endrule
```

## 子情况 2：多子模块方法调用 ✨新增

**bsc 输出**：
```
Error: "SdCtrl.bsv", line 100, column 8: (G0004)
  Rule `do_fsm' uses methods that conflict in parallel:
    spi.start(...)
  and
    cmd.build(...)
```

**原因**：同一 rule 内调用了多个不同子模块的 Action 方法（如 spi.start + cmd.build + resp.select）。BSC 将 rule 体内的所有方法调用视为并行冲突，不论它们是否在互斥的 case 分支中。

**关键认知**：`descending_urgency` 和 `mutually_exclusive` 无法解决 rule 内部的 G0004。

**解决**：拆为独立规则，每个规则只调一个子模块。

```bsv
// 错误：一条 rule 调了 spi + cmd + resp
rule do_fsm (state == ST_INIT && spi.busy == 0);
    spi.start(val);          // spi 子模块
    cmd.build(idx, arg);     // cmd 子模块
    resp.select(T_R1);       // resp 子模块
    state <= ST_NEXT;
endrule

// 正确：拆为两条，用 wait state 桥接
rule do_init_spi (state == ST_INIT && spi.busy == 0);
    spi.start(val);
    state <= ST_INIT_CMD;
endrule
rule do_init_cmd (state == ST_INIT_CMD);
    cmd.build(idx, arg);
    resp.select(T_R1);
    state <= ST_NEXT;
endrule
```

> **规则**: 同一 rule 内每个寄存器只能在一个无条件路径上被写入。复杂 FSM 中不同子模块的方法调用需拆到独立规则中。详见 `lookup_ref(topic="schedule")`。
