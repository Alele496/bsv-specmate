# AI 编码助手如何真正提升代码质量？—— 四场对照实验的方法论与意外发现

> 这是一篇技术博客草稿。记录的是我们在 BSV 硬件描述语言上做的一系列 AI 编码实验。
> 实验结论可能对任何在垂直领域使用 AI 编码的团队有参考价值。

---

## 问题

AI 编码助手（Copilot、Cursor、Claude Code）在主流语言上已经很好用了。但在**冷门语言/垂直领域**呢？

BSV（Bluespec SystemVerilog）是一门硬件描述语言，全球开发者可能只有几千人。没有 StackOverflow 上十万个问答，没有 GitHub Copilot 的训练数据。一个通用 Agent 写 BSV 代码时，连 Bool vs Bit#(1) 的区别都不知道。

我们做了一个 MCP 工具叫 **specmate**，嵌入了 BSV 领域的常见陷阱、编码范式和编译错误经验。但问题来了：**Agent 真的会用吗？怎么衡量它有没有用？**

---

## 实验设计

我们设计了四场对照实验，每场的方法论逐步升级：

### 第一战：RISC-V 外设（OpenCode 客户端）

- **4 个 Phase**：SPI Master → UART → BRAM → 跨时钟域 CDC FIFO
- **条件 A**：Agent 无 specmate，无任何编码规范
- **条件 B**：Agent 有 specmate（coding_rules + lookup_ref + preflight）
- **指标**：修复轮数、Token 消耗、编译通过时间

### 第二战：SD 卡控制器（CCB 客户端，/goal 自动循环）

- **1 个大模块，7 个子模块**
- **条件 A**：6 条固定静态规则
- **条件 B**：specmate + Supervisor 审查角色
- **新增**：把 specmate 的位置从"工具列表"改为"Supervisor 角色描述"

### 第三战：CRC-32 数据包处理器

- **新增双盲代码评审**：代码交给独立评委（不知道哪份是 specmate 辅助的）
- **指标**：代码质量评分（0-25）+ 编译通过时间 + 修复轮数

### 第四战：跨时钟域 SoC（3 级干涉强度）

- **首次引入 3 级干涉强度**：silicon（问什么答什么）/ wafer（主动建议）/ tapeout（话痨模式）
- **独立出题委员会**命题 + **独立评委**盲审（消除实验者偏见）
- **5 个子模块全部无法编译**（因为 BSV 编译器不支持跨时钟域）——但盲审仍然拉开了 11 分差距

## 关键发现

### 发现 1：silicon 社恐模式赢了话痨模式 8.5 分（第四战）

| 条件 | 盲审评分 |
|------|---------|
| 无 specmate | 85.5 |
| silicon（问什么答什么） | **96.5** |
| wafer（主动建议） | 88.0 |
| tapeout（话痨） | 88.0 |

这个发现是**反直觉**的。当前 AI 编码工具的主流设计思路是"给 Agent 越多上下文越好"——Cursor 拼命注入上下文，Copilot 拼命增强 prompt。但我们的数据显示：**给得太多反而有害**。

评委的评语一针见血：tapeout 模式的"你可能还需要考虑…"式建议，让 Agent 在次要问题上消耗了注意力，反而忽略了核心设计问题（CDC 缺少守卫条件、UART 缺少时序阶段描述）。

### 发现 2：Agent 不会自己想到调工具（全部四场反复验证）

"用 specmate" 这三个字写在 AGENTS.md 里 → **四场实验，Agent 全程 0 次调用。**

只有当 AGENTS.md 里写了**具体的调用时机和场景**（"收到 BSV 代码后先调 check_style → 有错误码问 guide(on_error) → 不确定的设计先查 lookup_ref"），Agent 才开始大量调用。

这意味着：**MCP 工具被发现（Agent 知道有这个工具）≠ 被使用（Agent 判断这个工具对当前任务有帮助）**。

### 发现 3：specmate 的核心价值是预防，不是修复（第一战）

第一战 Phase 2（UART）中，B 组在编码前调了 pre_code → specmate 提示用 `mkSizedFIFO` 而非自造环形缓冲区。这个建议发生在**写代码之前**，省掉了后续的架构级翻车。

所有实验中最高的 ROI 动作都是**编码前**的知识注入（pre_code 查陷阱/规范/参考文档），而非编码后的静态检查。

### 发现 4：代码质量 ≠ 编译通过率（第四战）

第四战 5 个子模块全部无法编译（因为 BSC 编译器不支持跨时钟域），但盲审仍然给出了 11 分的差距（96.5 vs 85.5）。评审从**结构清晰度、FSM 设计合理性、FIFO 使用规范、代码可读性**等维度评分。

这说明：编译通过率是一个过于粗糙的指标。AI 编码助手真正的价值体现在"工程性"——代码模块化、命名规范、设计模式选用——这些在编译前就能评估。

### 发现 5：7/7 全过的秘诀不是工具，是角色分离（第二战）

第二战 B 组 7/7 子模块全部编译通过，编码时间 -47%，Token -23%。很多人看到这个数据会说"specmate 真好用"。但仔细看日志：B 组 Agent 调用了 10+ 次 specmate 工具——这个调用频率来自**AGENTS.md 里的 Supervisor 角色描述**，不是来自 specmate 本身的功能。

角色描述 = "你是 BSV 专家。specmate 是你随时可以请教的领域搭档。"这个 3 行话的效果，远超"specmate 工具列表"的效果。

## 方法论总结

如果有人想做类似的实验，以下是我们的方法论框架：

1. **双盲评审不可省略**。任何知道"这份代码是用工具 X 写的"的评委，都无法做出无偏判断。第三战开始引入的双盲评审是实验可信度的分水岭。

2. **独立出题委员会**。让设计实验的人来命题，会不自觉地出"对自己工具有利"的题。第四战的外部出题委员会解决了这个问题。

3. **多级干涉强度**。只对比"有 vs 没有"等于没做实验。第二战如果只测"specmate vs 无"，结论就是"有用"。但第四战测了三个干涉级别，才发现"给得少反而好"——这个发现如果没有多级对比永远不可能得到。

4. **编译通过率不是唯一指标**。第四战证明了即使在代码都无法编译的情况下，盲审质量评分也是有效的差异化指标。

5. **工具调用日志必须分析**。四场实验反复揭示了一个模式：Agent 不会主动调工具。如果不分析日志，你会把一个"Agent 根本没用过"的工具当成"用了没效果"。

## 对 AI 编码助手的启示

这些实验虽然是 BSV 特化的，但几个结论可能对通用 AI 编码工具也有参考价值：

- **"更多上下文"不等于"更好"**。Agent 的注意力是有限资源。在正确时机给正确信息 > 一直注入信息。
- **工具的可发现性 ≠ 可调用性**。MCP 工具被发现后，Agent 不会自己判断"这个场景该用它"。需要更聪明的触发机制——可能是 role-based prompting，可能是客户端 hook，可能是更细粒度的 context-aware 建议。
- **预防 > 修复**。最贵的 bug 是架构级翻车。编码前的一次正确引导，价值远超编码后的十次静态检查。
- **双层 Agent 架构的成本收益**。Coder（无工具）+ Reviewer（有工具）的双层模式效果很好，但多 Agent 编排引入了 context 切换开销。单 Agent + Supervisor 角色描述在很多场景下性价比更高。
