# 议会：静态检查架构与 specmate 方向收敛

> 日期：2026-07-18 | 议题发起：SK | 参与者：specmate 负责人（技术评估）+ 顾问（战略分析）

## 起因

SK 发现 G0004 `case(reg)` FSM 模式误报，修复后质疑：项目里有 tree-sitter AST 解析器（`ast_query.mjs`），为什么 `check_style.mjs` 的 G0004 修复还在用正则？两条路线为什么完全割裂？

## 现状分析

specmate 负责人对代码做了全面审计，核心发现：

```
check_style.mjs（19条规则）
├── always-on（11条）：全部正则
├── full-scan（8条）：全部正则
        │
        └── 从未 import ast_query.mjs ❌

ast_query.mjs（tree-sitter）
├── findConflictPairs() — 已经有 WAW 检测
├── analyzeRuleConflicts() — 跨 rule 冲突
├── analyzeScheduling() — 调度评估
        │
        └── 从未被 check_style.mjs 调用 ❌
```

G0004 误报不是个案——是系统性的架构问题。

## 五个方案评估（specmate 负责人）

| 排名 | 方案 | 分数 | 核心逻辑 |
|------|------|:----:|---------|
| **1** | **C — BSC 作为终极校验** | **14/15** | Q3 已有 `check(compile=true)`，改 50-80 行让结果标 `bsc_verified: true/false`。BSC 零误报，静检退位为快筛 |
| **2** | **B — 混合方案** | **13/15** | 只改 G0004、arg count、multi-submodule 三条最高风险规则，~200 行，其他保持正则 |
| 3 | D — 置信度系统 | 11/15 | 不改规则，加 confidence 分，BSC 反向校准，长期收敛。但不提升规则本身精度 |
| 4 | A — 全 AST 升级 | 11/15 | 5 条规则的改造成本与精度收益不成比例 |
| 5 | E — 从 BSC 源码提取规则 | 7/15 | 跨语言，不可行 |

**负责人建议的路线**：

```
Week 1-2: 方案 C（BSC 验证输出）
Week 2-3: 方案 B（混合 3 条规则）
Week 4+:  方案 D（置信度做长期校准层）
```

一句话：**别和 BSC 比精度——把 BSC 变成 specmate 的后端。静检做快筛，编译做权威。**

## 顾问的战略分析

顾问独立评估后提出了一个更尖锐的问题：

> specmate 做了三件完全不同的事：静态检查、知识检索、BSC 编译对接。这三件事的价值密度完全不同。

| 能力 | 价值密度 | 可替代性 |
|------|:--------:|:--------:|
| 静态检查 | **中低** | 高（BSC 本身更准） |
| 知识检索 | **高** | 极低（没有其他 BSV 知识引擎） |
| BSC 编译对接 | **最高** | 低（BSC 只报错不解释） |

顾问的判断：specmate 应该**收敛到"知识引擎"方向**——收缩静态检查到"BSC 不报的语义陷阱"（约 6-8 条），主力投入到知识积累闭环。静态检查不是必要功能，它是门面——误报多的门面破坏信任，精准的门面建立信任。

**顾问的三层建议**：

| 步骤 | 做什么 | 投入 |
|------|--------|------|
| **立刻** | 19 条规则加 `confidence` 字段（high/medium/low），Agent 模板教 Agent 分级对待 | 极低，不改规则逻辑 |
| **近期** | 5 条 BSC 原本就会报的规则降级为 `info`，消息标注"BSC 编译会最终确认" | 极低，改 severity 字段 |
| **战略** | 不重写规则。把时间投到 TRAPS 扩充（12→30+）、DECISIONS 覆盖面（10→20+）、未知错误 LLM 诊断改善 | 中，但方向正确 |

## 议会共识（双方独立得出，高度一致）

| 结论 | 负责人 | 顾问 |
|------|--------|------|
| **不改 8 条 full-scan 为 tree-sitter** | A 方案排第四，ROI 不成比例 | "花 100 小时在规则精度上是 ROI 最低的选择" |
| **加 confidence 分级** | D 方案 11 分，做长期校准层 | "立即做，不改任何规则逻辑，只改 Agent 如何使用结果" |
| **BSC 作为地面真相** | C 方案排第一，14 分 | "Q3 orchestration 已改变游戏规则" |
| **护城河是知识积累** | "specmate 的真正护城河是知识积累闭环" | "花在 TRAPS 扩充、diagnose 改善上的价值更高" |

## 决议

1. **方案 C：BSC 作为终极校验**（排第一，14/15）
   - Q3 已有 `specmate_check(compile=true)` 直接跑 bsc 编译
   - 改 50-80 行让 check 结果标注 `bsc_verified: true/false`
   - 静检 = 快速反馈（秒级），BSC = 权威校验（分钟级）

2. **加 confidence 分级，立即做**（双方共识）
   - 19 条规则各标注 `confidence: 'high' | 'medium' | 'low'`
   - high: BSC 覆盖不到或不精确 → Agent 必须重视
   - medium: BSC 也能报 → Agent 参考
   - low: BSC 100% 准确覆盖 → 降级为 info，附带标注"BSC 编译会最终确认"

3. **5 条 low 置信度规则降级为 info**（近期）
   - P0073（重复类型参数）、P0085（重复属性）、G0030/G0040（urgency 循环）、G0054（未定义规则引用）、T0080（函数参数数量）

4. **不重写正则规则，战略聚焦知识积累**（长期方向）
   - 不改正则→tree-sitter，ROI 太低
   - 精力投入 TRAPS 扩充（12→30+）、DECISIONS 覆盖面、未知错误 LLM 诊断改善

## 核心信念

**specmate 的护城河不是代码分析精度，是 BSV 领域知识。别在规则精度上跟 BSC 较劲——把 BSC 变成后端，把精力放回知识积累。**
