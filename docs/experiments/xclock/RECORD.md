# 跨时钟域 SoC 子系统 — 实验记录

## 测试信息
- 测试时间：2026-07-05
- 客户端：CCB (Claude Code Best)
- 模式：goal 自主循环，auto 权限
- bsc 版本：2025.07
- 出题：独立 AI 考试委员会
- 盲审：独立 AI 按评分细则打分

## 实验设计

四个条件同时进行：

| 条件 | specmate LEVEL | 说明 |
|------|---------------|------|
| A | 无 | 对照组 |
| B1 | silicon | 社恐模式（问什么答什么，绝不多说） |
| B2 | wafer | 日常模式（该提醒的提醒，不多不少） |
| B3 | tapeout | 话痨模式（全程预警 + 追踪 + 追问） |

任务：5 模块 SoC 子系统（跨时钟域设计）。
各条件 0/5×3 轮编译机会。

## specmate 工具调用观察

| 条件 | R1-R2 调用 | R3 后调用 | 说明 |
|------|-----------|----------|------|
| B1 (silicon) | 0 | 激活 | R3 goal 写具体步骤后才调用 |
| B2 (wafer) | 0 | 激活 | 同上 |
| B3 (tapeout) | 0 | 激活 | 同上 |

"用 specmate"三个字无效——必须在 goal 中写出具体工作流步骤才能激活工具调用。

---

## 八个关键发现

1. **"用 specmate"三个字无效** — R1+R2 中 B Agent 0 次调用。R3 给具体步骤（"先调 preflight → 写代码 → check_style → guide(on_error)"）后才全部调用。Agent 不会自己想到工具——需要工作流指令。

2. **社恐模式赢了** — B1（silicon，话最少）以 96.5 分夺冠，比 B3（tapeout，话最多）高 8.5 分。话少 = 专注核心设计。silicon 只在 Agent 主动提问时才回答——Agent 问的都是它真正需要的。

3. **话痨拖后腿** — B3（tapeout）仅比 A 高 2.5 分。CDC 方法缺少守卫条件、UART 缺少时序阶段描述——恰是 B1 做对的部分。过多"你可能还需要考虑…"反而分散注意力。

4. **编译 0% ≠ 代码质量无差异** — 四套全部编译不过，但盲审跨度 11 分（85.5–96.5）。代码架构、设计思路、BSV 规范性的差异在盲审中清晰可见。

5. **specmate_guide 利用不足** — Agent 调 `check`（静态检查）多、调 `guide(on_error)`（错误诊断）少。错误码诊断路径被打断，同一个坑多轮修复而非查历史方案。

6. **Agent 不会自己想到调 specmate** — 四场实验反复验证。需要在 goal 里写工作流步骤。等 Agent 形成习惯后（通常 2-3 轮），调用频率自然上升。

7. **独立出题 + 独立盲审有效** — 考试委员会出题避免实验者偏差。独立盲审确保评分公正（评分 Agent 不知道条件归属）。此方法论应成为后续实验标准配置。

8. **Ultracode 多 Agent 可行** — Coder（无 specmate）+ Reviewer（有 specmate）工具隔离 + 自动循环验证通过。多 Agent 编排是可行协作模式。

## 下一场实验改进

Goal 中写：

> `specmate 是你的 BSV 编码搭档。写前问它陷阱，写完让它检查，报错让它诊断。`

把 specmate 描述为**伙伴**（partner）而非工具列表（toolbox）。
